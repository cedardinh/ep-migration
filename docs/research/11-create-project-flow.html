Create Project 端到端实现流程图

基于 docs/research 汇总的大颗粒度实现过程。第一版目标是前端请求进入后端，后端通过 web3j 向 Docker Besu 提交 TopazLifecycle.createProject 交易，节点接受后立即返回 txHash，失败时通过统一日志定位提交、nonce 或 RPC 问题。

Docker Besu RPC: http://127.0.0.1:8546 chainId: 1337 gasPrice: 0 不等待 receipt 不做 filters/event 订阅

一眼看懂

前置脚本稳定角色钱包、部署、授权矩阵，保证合约地址和业务 signer 的角色完整。
HTTP 层接收 JSON、生成 correlationId、返回 txHash 或稳定错误响应。
Gateway 层唯一处理 ABI 编码、签名入口、交易发送。
Nonce 层单 signer 串行分配 nonce，nonce 错误 reset 并重试一次。
日志出口异常统一到 ChainCallReporter 打印，主流程不散落日志。

链上执行与事件

交易出块后合约可能 emit ProjectCreated 和 ProjectStatusChanged，第一版接口不等待、不解析。

成功路径

1

请求进入

Controller 生成 correlationId，Service 做轻量校验。

2

ABI 编码

Gateway 把 DTO 编成 createProject(input) calldata。

3

签名发送

NonceManager 串行分配 nonce，本地签名，发送 raw transaction。

4

节点接受

Besu 返回 transactionHash，Reporter 可打印 CHAIN CALL SUBMITTED。

5

立即响应

HTTP 返回 txHash，不返回 projectId、status、blockNumber。

错误与日志链路

本地错误格式校验失败

  1. Service/Gateway 发现 address 或 bytes32 格式错误。
  2. 抛出本地应用异常。
  3. RestExceptionHandler 返回 400。

提交失败节点拒绝或 RPC 失败

  1. eth_sendRawTransaction 返回错误或请求失败。
  2. Gateway 分类为 nonce、RPC 不可用或节点拒绝。
  3. typed exception 携带 ChainCallContext。
  4. Reporter 打印诊断日志，HTTP 返回 502/503。

Nonce 错误提交阶段自愈

  1. eth_sendRawTransaction 返回 nonce 类错误。
  2. NonceManager 从 PENDING 重载 nonce。
  3. 只重试一次。
  4. 仍失败则作为链提交异常输出日志并返回 503。

RPC/节点错误不可用或拒绝

  1. 连接失败、超时、节点拒绝 raw transaction。
  2. Gateway 抛出提交异常。
  3. RestExceptionHandler 调 ChainCallReporter。
  4. HTTP 返回 502 或 503。

事件与 Receipt 边界

当前 createProject API 的成功语义是 transaction submitted，不是 project created。所以第一版不等待 receipt，不解析 ProjectCreated / ProjectStatusChanged，也不做 Besu filters、event subscription 或后台同步。
A

现在做

返回 txHash，日志里记录 submitted、nonce、from、to、correlationId。

B

链上随后发生

交易被打包后，合约成功时才会产生事件。revert 会回滚 event logs。

C

以后再做

如果产品需要展示最终状态，再单独建设 receipt/event 查询或同步模块。

D

不要混淆

不要把 submitted 日志写成 created，也不要从失败交易里找 event 错误消息。

E

排错来源

失败排查依赖 RPC message、nonce、transactionHash、ChainCallContext 和 correlationId。

F

请求关联

所有错误和提交日志都用 correlationId 串起来。
