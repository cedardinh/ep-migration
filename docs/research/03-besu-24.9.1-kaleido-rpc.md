# Besu 24.9.1 与 Kaleido 调研报告：后端如何调用链服务

## 推荐结论

当前项目最合适的接入方式是：

```text
Spring Boot backend -> web3j 4.9.8 -> Docker Besu JSON-RPC -> TopazLifecycle
```

理由：

- Docker 中已经运行 Besu 24.9.1，和目标文档版本一致。
- web3j 已经是后端依赖，不需要新增 SDK。
- 后端自己签 raw transaction，符合 Besu 文档建议，也不依赖 Kaleido 专有 REST gateway。

Kaleido 的 REST API Gateway / Eth Wallet Service 只作为调研背景保留，不作为第一版主路径。

## 当前 Docker Besu 链

本机已确认当前目标链：

```text
container: ep-besu
image: hyperledger/besu:24.9.1
host rpc: http://127.0.0.1:8546
container rpc: 8545
eth_chainId: 0x539
chainId: 1337
net_version: 2018
eth_accounts: []
clientVersion: besu/v24.9.1/linux-aarch_64/openjdk-java-21
```

后端、Hardhat `localhost` 网络和部署脚本都应使用宿主机 RPC `http://127.0.0.1:8546`。由于 Besu 不提供节点托管账户，部署脚本和后端交易都必须用配置文件里的私钥本地签名。

注意 `chainId` 和 `net_version` 不一致：

- `eth_chainId = 0x539 = 1337`，这是 EIP-155 replay protection 用的值，签名 raw transaction 必须用 `1337`。
- `net_version = 2018`，这是 networkId，与交易签名无关，不要拿它当 chainId 用。

这是 Besu `--network=dev` 的已知现象（networkId 与 chainId 取值不同）。已用 `curl eth_chainId` / `net_version` 在本机实测确认。后端配置和 Hardhat `localhost.chainId` 都用 `1337`。

## Besu API 基础

Besu 支持的 API 接入方式：

- JSON-RPC over HTTP
- JSON-RPC over WebSocket
- JSON-RPC over IPC
- RPC Pub/Sub
- GraphQL

默认端口：

- HTTP JSON-RPC：`8545`
- WebSocket：`8546`
- GraphQL：`8547`

重要默认行为：

- JSON-RPC 端点默认不启用，需要通过 Besu 启动参数开启。
- HTTP 默认 host 是 `127.0.0.1`。
- 如果需要远程访问，需要把 host 设为 `0.0.0.0` 并配置网络防火墙、安全组或反向代理。
- `--host-allowlist` 是防 DNS rebinding 的保护，不等于权限控制。

后端接入最少需要 Besu 开启 `ETH` API。常见调试和运维还会需要 `NET`、`WEB3`。

## Besu 不做账户管理

Besu 明确不支持 account management：

- 不实现 `eth_sendTransaction`。
- 应用应使用 `eth_sendRawTransaction` 发送已签名交易。

这直接影响后端设计：

- 后端不能把未签名交易交给 Besu 让节点帮忙签。
- 后端需要自己持有或接入签名能力。
- web3j `RawTransactionManager` 和 `Credentials` 是当前技术栈内最自然的实现方式。

## createProject 需要的 JSON-RPC 方法

最小调用链：

1. `eth_chainId` 或配置固定 chainId
   - 用于 EIP-155 replay protection。
2. `eth_getTransactionCount`
   - 获取 signer nonce。
3. `eth_sendRawTransaction`
   - 发送已签名交易。
4. 返回 transaction hash。

本项目当前不做 gas price 策略，交易 `gasPrice` 固定设置为 `0`，不调用 `eth_gasPrice`。

可选 view call：

- `eth_call` 调 `hasRole(PROJECT_OFFICER_ROLE, signer)` 做启动期健康检查。

Hardhat 只负责合约编译和脚本运行。第一版目标链不是 `npx hardhat node`，而是 Docker Besu `http://127.0.0.1:8546`。

## Revert reason 边界

对本项目的实践含义：

- 当前请求的成功语义只到 `eth_sendRawTransaction` 被节点接受。
- 如果交易后续链上执行 revert，第一版不会在当前 HTTP 请求内解析原因。
- `gasPrice` 直接传 `0`。
- 交易提交成功后立即返回 tx hash，不等待 receipt。
- 如果后续需要同步暴露合约 revert 原因，应单独设计 receipt/event/trace 或查询能力，不放入第一版 createProject 提交流程。

## Besu 24.9.1 特别注意

Besu 24.9.1 release notes 中与应用接入相关的点：

- 24.9.1 默认启用 receipt compaction，且启用后不能降级到 24.5.1 之前版本。当前 createProject API 不读取 receipt，但链数据目录兼容性仍需注意。
- `eth_createAccessList` 对 block number/tag 参数的处理有 bug fix。
- `admin_peers` 增加 `inbound` 字段。
- 新增部分 Engine API，不影响普通合约调用。

对 createProject 这类普通 JSON-RPC 合约交易来说，24.9.1 没有改变基本调用模型：后端本地签名 raw transaction，发送到 Besu，并在节点接受后返回 tx hash。

## Besu 权限和私有网络 API

Besu private network 文档包含 IBFT/QBFT/PERM 等接口：

- IBFT/QBFT 用于共识节点管理。
- PERM 用于本地 permissioning，且默认不启用。

createProject 不需要这些专有 API。只有当链环境启用了 account allowlist 或 node allowlist，才需要在部署/运维阶段确认后端 signer 或节点是否被允许。

## Kaleido 运行时接入方式

Kaleido 常见有三条路径：

1. Native JSON/RPC
   - 使用节点或网关提供的 Ethereum JSON-RPC URL。
   - 适合 web3j、ethers、viem 这种通用客户端。
   - 生产上通常通过 Kaleido App Credentials 做 Basic Auth。
2. REST API Gateway / EthConnect
   - 从 Solidity source/devdocs 生成 REST contract API。
   - 支持同步模式和异步模式。
   - 异步模式提供可靠交易队列、nonce 管理、receipt store、吞吐平滑。
   - 会绑定 Kaleido 的 gateway 模型，可移植性弱于 JSON-RPC。
3. Eth Wallet Service
   - Kaleido 托管交易签名 key。
   - DApp 可以请求服务签名交易。
   - 适合不希望后端直接持有私钥的组织策略。

## Kaleido 平台 API 与 App Credentials

Kaleido 平台 API 提供管理 app credentials 的接口，例如：

```text
/api/v1/consortia/{consortia_id}/environments/{environment_id}/appcreds
```

管理平台资源通常用 Kaleido API key bearer token。实际访问节点/gateway 时，Kaleido 文档和帮助中心指向 App Credentials，也就是运行时 Basic Auth。

当前实现使用 Docker Besu，不需要 Kaleido App Credentials 或 Basic Auth。以上内容只作为后续切换到 Kaleido 时的背景资料。

## 是否采用 Kaleido REST Gateway

REST Gateway 的优点：

- 自动生成 contract REST API。
- 可选 async 模式，内建 nonce 管理和 receipt store。
- 对业务系统看起来像普通 REST。

REST Gateway 的代价：

- 绑定 Kaleido 平台能力，切换回普通 Besu JSON-RPC 时需要替换。
- 需要额外维护 gateway 配置、OpenAPI、wallet/from 地址。
- 与当前后端已有 web3j 依赖重叠。

本项目目标强调“行业通用、可移植、强大不容易出错”。因此第一版建议用通用 JSON-RPC + web3j wrapper；只有明确要求托管密钥、高吞吐异步队列或 Kaleido 标准 gateway 时，再把 Kaleido REST Gateway 作为另一个 adapter。

## 推荐 Docker Besu 配置项

后端建议配置：

```yaml
ep:
  chain:
    rpc-url: http://127.0.0.1:8546
    chain-id: 1337
    lifecycle-contract-address: "0x..."
    signer-private-key: "0x..."
```

当前方案阶段约定：后端私钥直接配置在 `application.yaml`；部署私钥后续通过部署配置文件配置。
